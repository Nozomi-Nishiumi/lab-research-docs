PART 01

3カメラで跳躍の空中3次元姿勢を計測する

連携したAIを実際に使って、カエルなどの跳躍を対象に、3台のカメラから空中での3次元姿勢(四肢・体幹の関節配置)を復元するシステムを組む提案です。技術的な肝は「速く弾道的に動く小さな対象を、複数視点で"止めて"三角測量できるか」に尽きます。

3台を同一平面に置かず、異なる方位・仰角から空中ボリュームを見込ませる。視野が十分重なり、基線角が確保できる配置が三角測量の精度を決める。

最初に効く現実的な制約 ── カエルの空中フェーズは短く高速です。姿勢を解像するには目安として 100–250 fps 以上・露光は数ミリ秒以下・グローバルシャッターが欲しい。一般的な UVC webカメラは 30(一部60)fps・ローリングシャッターで、しかも3台をハードウェア同期できません。そこで本提案は「webカメラ3台=パイプライン構築と校正・低速検証の段階」と位置づけ、本計測は外部トリガ同期のマシンビジョンカメラ(以前評価された MARS1300 クラス)へ移行する二段構えにします。ソフトウェア資産はそのまま引き継げます。

開発プロセス(段階提案)

各フェーズは前段の成果物を引き継ぐ実際の順序

  1. 要件定義

    対象種と跳躍スケール、必要キーポイント(吻端・体幹・四肢関節・指端・総排泄口など)、目標精度を実空間 mm と再投影 px の両方で決める。ここから必要 fps・露光を逆算。

  2. 開発環境

    VS Code + Python(venv/conda) に OpenCV・DeepLabCut または SLEAP・anipose を導入し、git リポジトリを作成。以降の実装で AI 連携がそのまま効く。

  3. 3カメラ配置とプロトタイプ

    まず webカメラ3台で、静止/低速ターゲット(校正ワンド・LEGO テストシーン)を使いパイプライン全体を一度通す。配置は上図の「非同一平面・基線角確保」を満たす。

  4. 校正

    ChArUco ボードで各カメラの内部パラメータ、続いて外部パラメータをバンドル調整で最適化。既知長ワンドで三角測量誤差を数値検証(reprojection error / 実長との差)。

  5. 同期の確立と限界の見極め

    全視野に映る LED ストロボを同期マーカーにしてフレーム対応を取る。ここで webカメラの限界(高速運動でのサブフレームずれ・被写体ブレ)を定量的に確認 → 本計測はハードウェアトリガ同期のカメラへ移行する判断基準にする。

  6. 2D姿勢推定

    各視点の映像でキーポイントをラベリングし、DeepLabCut / SLEAP を学習・評価。リアルタイム用途は DeepLabCut-Live も検討。視点ごとに安定した 2D 検出を確保。

  7. 3D再構成

    anipose で多視点三角測量+時空間フィルタをかけ、各キーポイントの 3D 軌道を出力。再投影誤差でクオリティチェックし、外れフレームを補正。

  8. 運動学解析

    離陸角・重心軌道・四肢関節角・空中での姿勢制御(角運動量)などを算出。図表化まで含めコードを git 管理し、再現可能な解析パイプラインとして固める。

連携したAIの活かしどころ

VS Code 上の AI(特に Cline のようなエージェント型)に、次のような具体的タスクを渡せます。

  • OpenCV での ChArUco 校正・バンドル調整スクリプトの生成とデバッグ
  • DeepLabCut / anipose の config・カメラ校正定義の記述と修正
  • 三角測量+フィルタの実装(3視点の .h5 から 3D 軌道を出す処理)
  • 離陸角・重心軌道・関節角の算出コードと、matplotlib / ggplot での可視化
  • git リポジトリ構成・README・関数のリファクタリングと命名整理
PART 02

追跡が困難な場合の代替案

DeepLabCut / SLEAP による部位追跡がうまくいかないと判断された場合の代替戦略です。カエルの身体は模様に乏しくテクスチャ情報が少ないため、ブラーを消してシャープに撮れても「その部位が画素のどこか」を一意に決めにくい、という失敗が起こり得ます。ここは撮影では解決できないので、手法側で対処します。

まず失敗の種類を切り分ける

原因ごとに効く対策が違う。対策の前にコマ単位で見極める。

時間的欠損

ブラー

高速運動で四肢が溶ける。

撮影で解決済み
視点的欠損

オクルージョン

ある視点で部位が隠れる。

多視点で吸収
空間的曖昧性

テクスチャ欠如

写っていても特徴が無く点を決められない。

本丸 ── ここを解く
要点 ── テクスチャが無いと、視点ごとに2Dを当ててから統合する方式(DLC/SLEAP+anipose 三角測量)は原理的に弱い。各視点の2Dが曖昧なので、曖昧な点を三角測量しても曖昧なまま。曖昧性を消すには、画素の外から情報を足すしかない。原理は本質的に次の2系統。
A
空間の情報を統合する — DANNCE(ボリューメトリック回帰)多視点特徴を3Dグリッドに統合し、3D CNN が3D姿勢を直接回帰
主軸に推奨

各視点で2Dを確定させず、多視点の画像特徴を3D空間に逆投影して統合する。ある視点でテクスチャが無く単独では部位を特定できなくても、複数視点の特徴が3D空間で交差する場所として、学習された全身の形状事前分布と合わせて確率的に決まる。曖昧な2Dを先に固定しないので、曖昧性が三角測量に伝播しない。テクスチャが乏しく形状で決めるしかない対象に、原理的に正しい。

論文上の強み ── 出力が3D点で model-free に近く、再投影誤差や既知長ワンドで透明に検証できる。テクスチャ欠如への頑健性と、キネマティクス計測としての妥当性(モデルバイアスを主計測に入れない)を両立。
コスト ── 3Dラベルの学習セットが要るぶん立ち上げは重い。

具体的な実装要領

前提と準備

同期済みの多視点動画(ハードウェアトリガ。PART 01 のカメラ同期に接続)と、内部・外部パラメータの校正が起点。OpenCV / anipose の校正結果(K・歪み・R・t)を DANNCE のカメラパラメータ形式へ変換する。世界座標系は接地面と重力軸が意味を持つように取る(離陸角・重心軌道の解釈のため)。3D CNN はメモリ消費が大きいので GPU は目安 VRAM ≥12GB。

パイプライン
  1. 校正の変換 ── 既存の校正を DANNCE 形式へ。ワンドで三角測量誤差を先に確認しておく。
  2. COM(重心)ネットワーク ── DANNCE は2段構え。まず COMfinder が各視点で動物中心を検出→三角測量して3D重心を得る。COM ラベルを少数用意して学習。
  3. 3Dキーポイントのラベリング ── Label3D GUI で同期フレームに多視点ラベルを付け、三角測量で3D化。カエルのランドマーク(吻端・眼・肩・肘・手首・指端・股・膝・足首・指端・総排泄口・尾端など)を定義。まず数百フレーム規模。
  4. ボリューム設計(最重要ハイパラ) ── 重心まわりに切り出す3D体積(mm)は、跳躍で四肢が最大伸展しても収まる大きさに。体積を大きくすると voxel 解像度が粗くなる(メモリ制約)。grid 解像度(例 64³/80³)とのトレードオフが本タスクの肝。
  5. 学習 ── ラット等の学習済み重みから転移学習でラベル負荷を軽減(形態は異なるが体積特徴は一部転移)。SDANNCE を使うと骨格拘束(pose-graph)が入り、テクスチャの薄い四肢の一貫性が上がる。
  6. 推論・後処理 ── 3D座標を出力し、時間平滑(必要なら関節角速度の上限)。欠損コマの扱いを明記。
  7. 検証 ── hold-out ラベルとの Euclidean 誤差(mm)、再投影誤差、既知長ワンド。誤差の空間分布(ボリューム内の均一性)も報告。
詰まりやすい点 ── ①フレーム同期が甘いと高速運動で3Dが崩れる(ハードウェアトリガ前提)。②体積 vs 解像度 vs VRAM の三すくみ。まず小さめ grid で通し、精度を見て上げる。③ラベル数が精度を律速。合成データ(手法B①)での底上げが効く。
B
形状の事前分布を足す — CGモデルの活用テクスチャに頼らず、シルエットと全身形状の整合で姿勢を拘束
補助・二次解析

テクスチャで点が取れないなら「カエルの身体はこういう形」という形状拘束を持ち込む。テクスチャ欠如という条件では、CGモデルの価値は前段より明確に上がる。使い道は2つ。

  1. 合成学習データの生成器。多様な姿勢・照明・視点でレンダリングし、正確なラベル付き画像で DANNCE / 検出器を学習。テクスチャの薄い対象こそ実写学習が難しく、ここが効く。
  2. 形状事前分布としての補助フィッティング/相互検証。DANNCE の3Dがなお不安定な部位を形状拘束で埋める、あるいは独立にフィッティングして DANNCE の3Dと突き合わせる(両者一致は査読で強い)。
主計測にはしない理由 ── 市販CGモデルは対象種の骨格ランドマークに較正されておらず、関節中心も体節比率もアーティストの造形。測りたい関節角はモデルバイアスに最も敏感なので、それを主計測に据えると系統誤差が結論に乗る。関節角の主計測は "データが決める" DANNCE 側に置く。

具体的な実装要領

① 合成学習データの生成(手法A・検出器の底上げ)
  1. ツール ── Blender + Python API(bpy)。以前のカメラ localization で使った CG マップ資産とも親和。購入モデルのリグ有無を確認し、無ければ Rigify か手動アーマチュア+スキンウェイトで四肢を可動化。
  2. 撮像条件の一致 ── 実カメラと同じ解像度・内部パラメータ・3視点の外部配置でレンダリング。実写に寄せるためセンサノイズや軽微な劣化を付与し、sim-to-real ギャップを詰める。
  3. ドメインランダム化 ── 姿勢(関節角を生物学的可動域内でサンプル)、照明(HDRI・強度・色)、背景、材質・テクスチャをランダム化。テクスチャに依存させないため、あえて見た目も多様化。
  4. 真値の自動生成 ── シーンを制御しているので、正確な3D関節座標を各視点へ投影し、2D/3Dラベルを自動生成(DLC / DANNCE のラベル形式で書き出し)。手作業ラベル不要で無限に増やせる。
  5. sim-to-real ── 合成で事前学習→少数の実ラベルで fine-tune。合成のみモデルが実 hold-out で機能するかで、生成の妥当性自体を検証。
② 形状事前分布としての補助フィッティング(二次解析・相互検証)
  1. 微分可能レンダリング ── PyTorch3D か nvdiffrast でシルエット/深度をレンダリングし、観測との差を姿勢パラメータへ逆伝播。
  2. 関節可動モデル ── 線形ブレンドスキニング(LBS)でアーマチュアの関節に沿って変形。リグ無しモデルは骨格+スキンウェイトの整備が必要。動物向けの枠組みとして SMAL / SMALify の設計も参考になる。
  3. シルエット抽出 ── 背景制御下で背景差分、または SAM 等のセグメンテーションで各視点の前景を取得。多視点シルエット=ビジュアルハルが姿勢を拘束する。
  4. 損失設計 ── 多視点シルエット IoU+アンカー部位の再投影(吻端・眼・総排泄口)+時間平滑(関節角速度)+関節可動域の事前分布+形状/スケール項。
  5. 最適化 ── Adam で姿勢パラメータを最適化し、coarse-to-fine(全体の回転・並進・スケール→四肢)。初期値は前フレーム+DANNCE の3Dキーポイントで与え、局所解と左右対称の曖昧性を回避(3視点+陰影+アンカーで対称性を解消)。
  6. 出力と位置づけ ── 関節角つき全身3D姿勢が得られるが、主計測はモデルバイアスを避けて DANNCE 側に置き、これは相互検証/補完に用いる。両者の一致を示せれば査読で強い。
AIの活かしどころ ── VS Code 上の Cline 等に、bpy のドメインランダム化レンダリングスクリプト、校正フォーマットの変換、PyTorch3D の微分可能レンダリング最適化ループの雛形を書かせられる。
注意 ── DANNCE / SDANNCE / PyTorch3D などは更新が速いので、着手時に現行のリポジトリ・バージョン・依存関係を確認すること。
